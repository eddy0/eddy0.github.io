摘要 hash —摘要不是加密

  • 摘要算法
    摘要算法是一种能产生特殊输出格式的算法。给定任意长度的数据生成定长的密文.
    摘要结果是不可逆的, 不能被还原为原数据.
    理论上无法通过反向运算取得原数据， 而且一个安全的摘要无法找到碰撞.

  • 作用:

    • 数据完整性验证, 下载的文件比对 sha1 的摘要结果， 就可以知道文件是否被篡改
    • 保护用户密码， 防止撞库， 用 A 网站的密码到 B 网站测试登录

  • 常用的算法 md5 (32 字符) 和 sha1 (40 字符), sha256

  • 问题: 彩虹表, 穷举各种组合的摘要.

加盐

盐（salt）是为了防止黑客对密码进行碰撞，假如用户使用简单密码, 破解者可以用提前生成的简单密码摘要表（彩虹表）来破解原文。所以我们会存储一个额外的信息，扰乱用户的简单密码。
加盐是在用户密码的基础上再加一些字符串， 然后再用摘要

  • 作用 : 让黑客拿到的数据是无意义的，让彩虹表无效.

重置密码的功能

用户点击忘记密码, 给客户邮箱发一个 url, 并且在 query 加一串随机字符 (自己设置伪随机字符串，比如 uuid,
不能是有规律的值, 比如用unix 时间)
客户点击之后, 发送 get 请求, 判断是否 expire (发送的时间, 过期时间和现在时间对比)
如果ok, 就跳到重置密码页面

证书

数据传输过程的任何环节, 层层路由器都可能被拦截和窃取. 用明文传输密码是绝对不安全的, 这就需要 https
https 是发数据的时候把页面所有的传输内容都加密了
https 的原理, 是非对称性加密, 把 http 的数据加密之后在浏览器和服务器之间传输.

  • 非对称性加密加密算法:
    rsa, 每个人都有一对公钥和私钥
    发数据的时候, 浏览器先用服务器的公钥加密, 公钥加密的数据只有私钥能解开, 所以服务器可以用自己的私钥解密, 服务器解密之后就可以拿到原始的 http 请求, 然后解析请求做响应. 其他任何人都无法不到原始信息.
    服务器把浏览器的公钥拿过来, 加密响应之后再发给浏览器, 只有浏览器的私钥才能解开.

  • 中间人攻击: 服务器发给服务器的请求被拦截, 导致请求并没有发送到服务器, 而是被中间拦截的人伪造服务器返回了, 而用 https 就可以避免这个情况, 因为中间人无法得到原始数据.

  • 证书: 浏览器和服务器都由证书信任机构来保证 https 的进行. 解决通信之间的信任问题, 由权威机构颁发证书确认这个 url 确实属于这个公司.

  • 对称性加密比较出名的是 aes
    加密的秘钥和解密的秘钥用的是同一个, 用于 session， 需要知道原来的信息

cookie

  • 原因 为了让服务器识别不同用户发的请求，一种方式是在 url 的 path 后面加一个标记, 比如 /?id=1, 这样就可以知道是哪一个用户, （1. 但是每个人都可以发这样的请求, 别人也可以伪造这样的 path 发请求, 无法知道是否真的是这个用户发的. 2. 这样所有的其他页面的path都要加上 id 才能识别了） 另一种方法就是 cookie, 就是用来确定用户身份, 利用 header 里面的 cookie 字段, 不需要在 path 里面加标记就可以识别的不同用户

  • 方法:

    • 服务器在 header 里面设置一个 Set-Cookie 字段 Set-Cookie: user=xxx, 发送给服务器
    • 服务器接收数据之后发现有 Set-Cookie 字段, 就会把 cookie 和内容保存到本地
    • 服务器下次发同一个 host 网页请求的时候, 会在 header 里面加上一个 Cookie 的字段 Cookie: user=xxx

    • 这样服务器接收的 header 里面就会有一个 Cookie 的数据, 解析之后可以与数据库的信息比对, 看是不是同一个用户

    • cookie 可以放在header 里面, 也可以放在 ETag 里面 (基本不用).

  • 优点 解决服务器识别不同身份问题

  • 问题 安全问题, 身份伪造, cookie 容易猜和模拟. 服务器识别用户是通过字符串的，类似于 id=1 这样的简单的cookie非常容易猜到, 这样别人可以在 header 里面模拟别人的 cookie 数据发请求, 所以 cookie 相对没有 cookie 的时候安全性高了一点, 但是还是无法解决身份伪造的问题, 这就需要用session 解决问题.

session –session 是一个理念

  • 目的: 为了解决 cookie 无法解决的身份伪造的问题.

  • 方法: 生成一个随机的字符串给 set-cookie, 这样就不容易猜其他用户的 cookie 信息了. 与 cookie 不同的地方在于： 服务器 parse header {id: xxx} 之后, 需要先与保存 session 的数据库进行比对 session 的保存顺序刚好颠倒一下: {xxx: id}, 得到这个值(xxx)的value, 再通过这个 key 去查询或者处理数据库中这个用户的其他信息.

  • 作用: 服务器通过随机字符串来验证身份， 防止身份伪造，主要是防止猜.

  • 问题: 无法防止 cookie 被盗的情况(比如信息被公共 wifi 劫持) — 解决方法: 加盐加密

  • session 持久化 (重启后仍然可以用)

    • cookie 保存到服务器的文件中, 请求的时候读取一下文件
    • 对称性加密, cookie 也可以存在客户端(通过对称性加密方式存, 服务器只要通过 key 把信息解析出来就好了)

  • session 共享 (sso) 主要用于是同一个公司不同服务器 (同一旗下不同的网站)共享 session 数据, 连接到网站服务器的时候先插一个 session 服务器, 如果没有登录就直接返回错误了

重定向

服务器在接受到一个需要重定向的 url 请求后, 在返回的请求头里 status code 设置为 301 (302), header 里面设置 Location: ${新url}, 返回给客户端
浏览器解析 header 发现 301 就会去 header 里面去找 Location 的 url, 自动重新再用 Location 的 url 发一次新的请求
